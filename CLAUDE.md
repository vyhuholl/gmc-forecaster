# gmc-forecaster

Прогноз спроса (ЗАКАЗЫ) в бизнес-симуляции Global Management Challenge.
Двухшаговая модель «доля рынка → спрос». Единица прогноза — **9 ячеек**:
3 продукта × 3 канала (EAEU/ЕАЭС, ASEAN/АСЕАН, INT/Интернет).

Почему именно share-модель, а не прямой own-demand, и весь дата-анализ,
стоящий за выбором, — в [docs/FINDINGS.md](docs/FINDINGS.md). Там же нюансы данных
(спрос ≠ продажи ≠ доля; доля эндогенна и течёт как признак).
**Важно:** docs/FINDINGS.md — исследовательский журнал, местами отстаёт от кода
(см. «Расхождения»). При конфликте ориентируйся на код.

## Запуск

```bash
uv sync                 # окружение
make validate           # ruff format + ruff check + mypy --strict + pytest
uv run gmc-forecaster --help
```

Три подкоманды: `forecast` (прогноз под решениями игрока), `backtest` (оценка
на истории) и `cost` (полная себестоимость + contribution margin по 9 ячейкам).

```bash
# «крутить цены/рекламу»: правишь решения на листе 'Your decisions' в --current
uv run gmc-forecaster forecast \
  --current  data/W115264.xls \
  --train    data/W1152*.xls data/W1312*.xls \
  --history  data/Hst*.xlsx --out forecast.csv

# оценка качества модели по смежным парам кварталов
uv run gmc-forecaster backtest \
  --reports data/W1152*.xls data/W1312*.xls \
  --history data/Hst*.xlsx --out backtest.csv

# себестоимость решения по 9 ячейкам (--train/--history — кварталы своей фирмы
# для калибровки yield/спот-цены сырья)
uv run gmc-forecaster cost \
  --current data/W115264.xls \
  --train   data/W11526[1-3].xls --out cost.csv
```

Решения (цены по 9 ячейкам, реклама, **пер-канальные** число/комиссия
дистрибьюторов EAEU/ASEAN/INT) читаются **прямо с первого листа
`'Your decisions'`** файла `--current` (`parser.parse_decisions`, позиционная
схема `DEC_SCHEMA`, языконезависимо) — **json-сценарий убран**. Рабочий процесс:
правишь свои what-if решения в этом листе Excel → `forecast`. Реклама сворачивается
в множитель бюджета компании-уровня; дистрибьюторы действуют на все 3 продукта
канала, наблюдаемы только у своей компании → дают множитель собственной
привлекательности `A_own` в логите доли. Сила эффекта дистрибьюторов — коэф.
`k_n`/`k_comm` (дефолт 0.15/0.10, опц. оверрайд флагами `--dist-k-n`/`--dist-k-comm`).
**Найм дистрибьюторов отложен на квартал**: выходит на охват лишь со следующего
за прогнозным квартала (Q_next+1), поэтому на прогноз Q_next дистрибьюторы **не
влияют** (цена/реклама — влияют сразу); их отложенный вклад выносится в отдельную
колонку **Δ_дистриб_след_%**. Квартал прогноза = следующий за кварталом в `--current`.
Прогноз **заякорен** на наблюдаемый спрос: `спрос_сцен = спрос_текущ ×
сезонность × рычаг`, где рычаг = модельное отношение доли (сцен/база),
демпфированное `--lever-k` (дефолт 0.5; 1 = полный рычаг, 0 = чистый сезонный
наив) и ограниченное клипом `[1/3,3]`. Наблюдаемый спрос несёт всю
firm×cell-гетерогенность, которую логит доли не восстанавливает → прежний
абсолютный `доля × объём` давал 15-20% MAPE (см. [docs/BACKTEST.md](docs/BACKTEST.md)),
заякоренный — ~8% (бьёт сезонный наив). Колонка **Δ_рычаг_%** = рычаг−1,
изолирует эффект решения (при неизменных решениях = 0 → спрос_сцен = сез. наив).
После таблицы прогноза `forecast` печатает **коэффициенты модели доли** со
значимостью (ст. ошибка, t, p, звёзды) + R²/n/edf/λ и строку смысла β_price.
Флаг `--coeffs {key,full,none}` (дефолт `key`: ключевые признаки + наклоны u_g;
`full` до-выводит все FE ячейка/группа/фирма; `none` выключает). При `--out`
таблица коэффициентов дублируется в sibling `<out>.coef.csv`.

## Архитектура (`src/gmc_forecaster/`)

- **parser.py** — `parse_report(path)` → `(meta, df по 9 ячейкам)`. Читает
  **только лист `W`** (плоский числовой экспорт) по позиционной схеме `SCHEMA`
  (0-based индексы, `off = 3*(product-1) + ch`). Не зависит от языка и версии
  отчёта. `_num`/`_stars` — нормализация ячеек/рейтинга-звёзд. Движок чтения —
  `engine="calamine"` (читает и старый BIFF `.xls`, и strict-OOXML `.xlsx`
  истории, где openpyxl не видит листов; тот же движок в `load_panel`).
  `parse_decisions(path)` → dict решений игрока с **первого листа
  `'Your decisions'`** (позиционная схема `DEC_SCHEMA`, тоже языконезависимо):
  `price{ячейка}`, `adspend_total` (тыс.ерз), `dist_n{канал}`, `dist_comm{канал}`;
  пустая ячейка опускается (наследует текущее из `W`). Заменяет старый
  `scenario.json`. `parse_costs(path)` → `CostReport` (dataclass) с производ-
  ственными и стоимостными данными из **листов `'Resources and products'` и
  `'Financial statements'`** по позиционным схемам `RES_SCHEMA`/`FIN_SCHEMA`
  (0-based `(row, col)`, продукты в столбцах 20/22/24; языконезависимо): выпуск/
  брак/полуфабрикаты по продуктам, часы сборщиков/станков, эффективность,
  расход/закупка сырья, все статьи накладных и P&L (зарплаты, эксплуатация
  станков, ОТК, транспорт, амортизация). Требует отчёт компании 1..8.
- **model.py** — ядро:
  - `load_panel(paths)` → панель по всем 8 компаниям (price/share/adspend/rating),
    + внешняя опция `share_out = 100 − Σ`.
  - `ShareModel` — **стадия 1**: логит доли через инверсию Берри
    `ln(sᵢ/s₀) = β·X + FE(ячейка) + FE(группа) + FE(фирма) + (β_price+u_g)·log_price`.
    Оценка — **штрафованный МНК** (штраф только на u_g), признаки
    `FEATURES = [log_price, log_adspend, rating]`.
    **Firm-эффект** (дамми company×group, центр. внутри группы) ловит устойчивую
    фирменную гетерогенность → снижает регрессию-к-среднему в долях; это уровень,
    не наклон → эластичность/`Δ_рычаг` почти не меняются. Незнакомая фирма
    откатывается на групповой эффект.
    **Random slope наклона цены `u_g`** — эластичность гетерогенна по группам
    (эмпир. −2.5…−4.9): `u_g` усаживаются к global (ridge), λ из
    leave-one-quarter-out CV (`ridge_`) → наблюдаемая группа берёт свой наклон,
    разреженная/новая (мало кварталов) откатывается к global (partial pooling).
    На текущих данных CV выбирает пулинг (свои наклоны OOS не обобщаются).
    `predict_shares` даёт MNL-замещение и **исключает неактивные компании**
    (`price≤0` — не предлагают продукт; иначе клип 0→1 дал бы фантомно дешёвого
    игрока, забирающего почти всю долю). `counterfactual` — доли при смене цены.
    `coef_summary()` → таблица коэффициентов со **значимостью**: ст. ошибки из
    сэндвич-ковариации штрафованного МНК `σ²·Ainv·XᵀX·Ainv` (для непенализуемых
    признаков = OLS `σ²·(XᵀX)⁻¹`), σ² на edf=tr(Ainv·XᵀX), p — нормальное
    приближение (`erfc`). Для наклонов `u_g` (штрафуются) значимость приблизит.
  - `fit_seasonality(hst)` → сезонные множители {Q:factor} по группе 0.
    `cell_volume` = own_sold/(own_share/100). **Стадия 2**: объём ячейки с
    сезонной поправкой (для калибровки/oracle-отката).
  - `damp_lever(ratio, k)` — демпфированный (к 1.0 на долю 1−k) и ограниченный
    клипом `LEVER_CLIP=[1/3,3]` причинный рычаг; `LEVER_K=0.5` — дефолт демпфера
    (общий для forecast/backtest). **Итоговый спрос заякорен:** `спрос_t ×
    сезонность × damp_lever(доля_сцен/доля_база, k)` — наблюдаемый спрос несёт
    firm×cell-гетерогенность, логит даёт лишь причинный сдвиг (см. BACKTEST.md).
- **forecast.py** — `forecast(current, train, history, k_n?, k_comm?, lever_k?)`:
  собирает пайплайн, читает решения игрока из `parse_decisions(current)` (лист
  `'Your decisions'`), применяет их к своей компании, ЗАЯКОРИВАЕТ прогноз на
  наблюдаемый спрос (спрос_база = спрос_текущ × сезонность; спрос_сцен = база ×
  рычаг). Возвращает df (спрос текущий/база/сценарий, доля, Δ_рычаг) +
  `df.attrs["meta"]` (в т.ч. `lever_k`), `df.attrs["coef_summary"]` (коэф. со
  значимостью) и `df.attrs["fit"]` (R²/n/edf/λ). `k_n`/`k_comm` — сила эффекта
  дистрибьюторов; `lever_k` — демпфер рычага (дефолты).
  **Режим 1-й итерации half-final** (`_forecast_history`, авто-переключение при
  `--current` = history-рамп: компания вне 1..8, своих долей нет — доли `'Not
  requested'`, 8 «компаний» листа `W` идентичные клоны): стадия 1 недоступна →
  бейзлайн = персистенция наблюдаемого спроса, рычаг решений = **заимствованная
  собственная эластичность** из модели на `--train` (регулярные отчёты с
  долями): `Δln(спрос) ≈ β_price·Δln(цена) + β_adspend·Δln(реклама+1) +
  ln(dist_mult)`, демпфер `lever_k`. Грубый перенос → `meta["mode"]=="history"`,
  `доля_сцен_%`=None; CLI печатает предупреждение. `--train` обязателен (даёт β).
- **backtest.py** — `backtest(reports, train, history, lever_k?)`: прогон по
  смежным парам кварталов (Q_t→Q_{t+1}) внутри серии (группа+компания),
  сравнение с фактом. Раскладывает ошибку на стадию 1 (доля, MAE) и стадию 2
  (объём, MAPE), **заякоренный сквозной спрос** (`_anchor_demand`: спрос_t ×
  сезонность × рычаг; MAPE) против бейзлайнов (персистенция, сезонный наив), в
  двух режимах: **realistic** (конкуренты заморожены на Q_t) и **oracle** (факт
  рынка Q_{t+1} целиком). Нет наблюдаемого спроса → откат на абсолютный
  доля×объём. Сценарий не нужен: решения берутся из отчёта Q_{t+1}
  напрямую. Возвращает `(сводка, детализация по ячейкам)`. `per_cell_summary(df)`
  раскладывает детализацию по 9 ячейкам (канал × продукт): доля-MAE (real/oracle),
  объём-MAPE, спрос-MAPE + бейзлайны — но без метрики «все 8 компаний» (в
  детализации только своя ячейка).
- **cost.py** — `cost(current, train?, history?, yield_halflife?)` → df по 9
  ячейкам (ед/цена/выручка, компоненты затрат, `себест_полн_ед`, `CM_ед`,
  `прибыль_полн_ед`) + `df.attrs` (meta/пулы/yields/сверка с P&L). Себестоимость
  строится **снизу вверх** из авторитетных констант мануала GMC (Табл. 5–18: сырьё
  1/2/3 шт, станко 60/75/120 мин, сборка 100/150/300 мин, машино-час 8, ОТК 1/ед,
  амортизация 2.5 % и т.д.) и рыночных величин отчёта (спот-цена сырья, зарплаты,
  комиссия), а не из строки COGS P&L (та смешивает движение запасов). Тождества
  сверены на данных: `сырьё = Σ(произв−п/ф)×{1,2,3}` (точно), `машино-часы =
  Σ(произв−п/ф)×{60,75,120}/60/эфф` (≤0.1 %; полуфабрикат замещает станко-этап).
  **Разнесение на 9 ячеек двухступенчатое** (производство по ПРОДУКТУ, продажа по
  КАНАЛУ): материал → по единицам; конверсия (зарплаты сборщиков/механиков,
  эксплуатация станков, амортизация, ОТК) — фикс-пул по **сборщико-минутам**;
  канал-пул (дистрибуция/транспорт/интернет/гарантия) → к каналам (интернет в INT,
  прочее по выручке), внутри канала по единицам; overhead (маркетинг + прочие
  накладные) → **по выручке**. Три yield-поправки калибруются по кварталам своей
  фирмы (EWMA, `calibrate_yields`): брак → делить на годные, болезни → конверсия/ед
  ↑, эффективность станков → станко-затраты ↑. Произв. фиксы **пересобираются по
  сменам** (`shifts_needed`/`reconstruct_machine_opex`; смены =
  `⌈доступ_ст-часы/(станки×576)⌉`), не берутся прошлым фактом. Два выхода: полная
  себестоимость (absorption) и **contribution margin** (`цена − материал − п/ф −
  комиссия`; конверсия — полу-фикс, только в полную). `unit_costs(...)` → лёгкий
  `{ячейка: {full, cm}}` для интеграции в `forecast`. Требует компанию 1..8
  (иначе `ValueError`; `unit_costs` → None). Инференс расхода МНК — лишь
  sanity-check (`infer_raw_coeffs`), не источник нормативов.
- **cli.py** — точка входа `gmc-forecaster` с подкомандами `forecast`,
  `backtest` и `cost` (entry point в pyproject). `forecast` после таблицы печатает
  блок коэффициентов (`--coeffs key|full|none`, `_print_coeffs`); при `--out` пишет
  sibling `<out>.coef.csv`; таблица дополнена колонками `себест_полн_ед`/`CM_ед`/
  `прибыль_ячейка` (при отсутствии производ. данных — пропуск с предупреждением).
  `backtest` с флагом `--per-cell` печатает 9-строчную таблицу метрик по ячейкам
  (`per_cell_summary`/`_print_per_cell`); при `--out` дублирует её в sibling
  `<out>.percell.csv`. `cost` печатает смету по 9 ячейкам + yield/сверку с P&L;
  при `--out` пишет CSV.

## Данные (`data/`)

- `W<gg><c><yy><q>.xls` — отчёт: группа/компания/год/квартал.
  `W1152*` = гр.11 комп.5 (Q1–Q4'26); `W131*` = гр.13 комп.1 (Q1–Q3'26);
  `W318*` = гр.31 комп.8 (Q2–Q3'26); `W496*` = гр.49 комп.6 (Q2–Q3'26).
- `Hst*.xlsx` — history-кварталы группы 0 (компании-клоны) → только сезонность.
- `1Quarter..5Quarter` = гр.1 комп.1 (2017–18); `6..10.xls` = гр.2 комп.4 (2018–19)
  — разные версии (RU/PL/ES), проверка версионной устойчивости парсера `W`.
  **Гр.1 не экспортирует доли** (позиции 331+ пусты) → непригодна для стадии 1;
  гр.2 годна (файл `10` — терминальный квартал, долей нет). У старых игр 2017–19
  нет своей сезонности (история группы 0 — 2024–26, неприменима).
- `gmc_w_schema.json` — выведенная схема листа `W` (справочно).

## Расхождения кода с docs/FINDINGS.md

- Имена `gmc_parser_flat.py` / `gmc_share_model.py` / `gmc_predict.py` устарели —
  это модули пакета выше + CLI `gmc-forecaster`.
- `scenario.json` **убран** целиком: решения читаются с листа `'Your decisions'`
  файла `--current` (`parse_decisions`); квартал — из `--current`.
- В коде остались парсеры листа `W` (`parse_report`) и первого листа решений
  (`parse_decisions`); обоих ведёт позиционная схема, именованного парсера нет.
- Ридер — единый `engine="calamine"`, а не автовыбор xlrd/openpyxl (§13): история
  сохранена в strict-OOXML, openpyxl её не читает (пустой список листов).

## Конвенции

- Python ≥3.14, `mypy --strict`, ruff line-length 79. Комментарии/вывод — на русском.
- Весь Python-код запускай только через `uv run`.
