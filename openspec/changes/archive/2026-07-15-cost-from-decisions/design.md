## Context

Подкоманда `cost` (change `add-cost-accounting`) считает себестоимость по 9
ячейкам снизу вверх. Но входы она берёт из ФАКТОВ прошлого квартала: объём
ячейки — `sold` (лист `W`), цены — `price` листа `W`, реклама/накладные — из
`Financial statements`. Для инструмента what-if (правишь решения → смотришь
себестоимость) это неверно — надо считать от РЕШЕНИЯ, которое подаётся.

Всё нужное лежит на первом листе `'Your decisions'` (его уже читает
`parse_decisions` для `forecast`). Разбор реального листа (`W115264`, 0-based):

```
строки 13-15  реклама (имидж c4, продукты c5/7/9) + агенты (c15) + вознагр (c19)
              + комиссия % (c22)          [цена/реклама/комиссия — уже в схеме]
строки 18-20  цены по каналам × продуктам (c5/7/9)      [уже в схеме]
строки 23-25  ПЛАН ПОСТАВОК по 9 ячейкам (c5/7/9)       ← новый вход (объём)
row19  c22    число рабочих смен = 1                    ← новый вход
row20  c15/22 интернет-порты / развитие сайта           ← новый вход
row23  c15    сборщики: найм(+)/увольнение(−)           ← новый вход
row24  c15    часовая ставка сборщиков = 10.5           ← новый вход
row25  c15/22 упр.бюджет / тренинг                       ← новый вход
row29  c5/7/9 R&D по продуктам = 25/25/25               ← новый вход
row30  c5/7/9 ВРЕМЯ СБОРКИ = 128/188/375                ← новый вход (см. Реш. 2)
row30  c15/22 покупка / продажа станков                 ← новый вход
```

Две находки подтверждены на данных:

1. **Объём = `sold` неверен.** На `W115264`: продано 7383 > произведено 6781 >
   поставлено 6632 (распродажа склада). Материал на проданном → смета 2 564 053
   vs факт COGS+накл 2 403 324 (+6.7 %).
2. **Время сборки захардкожено.** `assembler_worked_h=19180` совпадает с
   `{128,188,375}` листа решений (19182 ч), а не с константой `{100,150,300}`
   (15152 ч). Станко-минуты `{60,75,120}` при этом верны (`4967` vs факт `4963`).

Ограничения проекта: Python ≥3.14, `mypy --strict`, ruff line-length 79,
комментарии/вывод на русском, весь код через `uv run`, движок `engine="calamine"`.

## Goals / Non-Goals

**Goals:**

- Себестоимость = стоимость производственного прогона, заданного РЕШЕНИЯМИ (план
  поставок, цены, реклама, комиссии, смены, персонал, станки, R&D), а не фактами
  прошлого квартала.
- Веса разнесения конверсии — на фактическом времени сборки из решений.
- Две базы объёма продаж (план по умолчанию; прогноз через `--forecast`).
- Сверка с отчётом сохранена как страховочный контроль (на предзаполненном листе).

**Non-Goals:**

- Оптимизация решения (подбор цен/выпуска под макс. прибыль).
- Точное воспроизведение бухгалтерской COGS с потоками запасов (мешает движение
  склада — считаем экономическую себестоимость прогона).
- Полностью проспективное моделирование полуфабрикатов (v1 — из отчёта).
- Прогноз затрат конкурентов.

## Decisions

**1. Объём ячейки = план поставок (строки 23–25), произведено = план × yield.**
План поставок = `planned` = `shipped` по 9 ячейкам (сверено). Производственные
пулы (материал, конверсия, ОТК) считаем на ПРОИЗВЕДЕНО = план, инфлированный
`defect_infl` (буфер брака) — как и было для перехода годные→произведено. Это
закрывает разрыв «продано vs использовано».

**2. Время сборки — из решений (row30), не из константы.** `ASSEMBLY_MIN`
`{100,150,300}` не соответствует игре (`assembler_worked_h` совпадает с
`{128,188,375}` листа). Читаем времена сборки с листа решений; константу оставляем
как фоллбэк при пустых ячейках. Станко-минуты `{60,75,120}` не трогаем (верны).

**3. Себестоимость/ед и CM/ед — на базе ПЛАНА, независимо от базы продаж.**
Полная себестоимость — это absorption производственного прогона (плановый выпуск),
она не должна «дышать» от прогноза продаж. Веса overhead-по-выручке берём по
ПЛАНОВОЙ выручке (`цена × план`). База продаж влияет только на РЕАЛИЗАЦИОННЫЕ
колонки (выручка, прибыль_ячейка, CM_итог), не на per-unit показатели.

**4. Две базы продаж — колонками; прогноз опционален через CSV.**
- База=план (дефолт): продано = план поставок.
- База=прогноз: `cost --forecast forecast.csv` читает `спрос_сцен` по (`канал`,
  `продукт`), клип `продано ≤ поставлено`; добавляет параллельные реализационные
  колонки. Ячейка отсутствует в CSV → откат на план.
Почему CSV, а не звать `forecast` внутри: `cost --train` (своя фирма, калибровка)
и `forecast --train` (конкуренты, доли) семантически разные — совмещать в одной
команде перегруз; CSV — существующий артефакт, читается без импорта `forecast` →
никакого цикла зависимостей.

**5. Интеграция в `forecast` — без изменений (упрощение при реализации).**
Выяснилось: раз себест/ед базонезависима (решение 3), `unit_costs` НЕ нужен
параметр `demand` — он возвращает базонезависимые `full/cm`, а
`forecast._add_cost_columns` уже умножает их на свой `спрос_сцен`. Поэтому
`forecast.py` и `unit_costs` не менялись; демонстрация базы прогноза для
standalone идёт через `cost(demand=...)` и CLI-флаг `--forecast` (CSV читается без
импорта `forecast` → цикла нет).

**6. Производственная экономика из решений — частично; труд-пул из отчёта.**
Реализовано из решений: число смен (row19 c22, fallback `shifts_needed`) и парк
станков (текущие ± покупка/продажа row30 → амортизация и `machine_opex`).
**Зарплаты сборщиков/механиков оставлены из отчёта** (`×absence_infl`):
реконструкция «штат × часы × ставка» на данных шумная (±10 % против факта), а
отчётная сумма точна и держит сверку. Следствие/ограничение: труд-пул не реагирует
на правку ставки/найма — фикс-пул уровня квартала (капасити-лычаг = смены, он
решенческий). R&D/сайт/прочие накладные — из отчёта (замена реализована только для
рекламы, где решение = факту сверено). Это v1: within-capacity корректно, крупная
правка объёма недо-нагружает труд (см. Риски).

**7. Сверка двухрежимная.** Предзаполненный лист = исполненные решения → смета
сходится с `COGS + накладные` этого отчёта (в пределах inventory-шума, т.к. COGS
считает ПРОДАННОЕ, а смета — ПРОИЗВЕДЁННОЕ; при распродаже склада расходятся
закономерно). Отредактированный лист (what-if) → форвард-оценка, сверка
помечается «неприменима». Признак «отредактированности» — расхождение плана
решений с `planned`/`shipped` отчёта.

## Risks / Open Questions

- **Полуфабрикаты.** `components_used` замещает станко-этап и сильно снижает
  расход сырья (на `W115264` использовано 4834 шт против 9548 по поставкам,
  `components_purchased=0` → делали из своих запасов). Проспективно из решений это
  не восстановить чисто (нет решения «сколько деталей сделать»); v1 берёт
  замещение из отчёта. Риск: при смене выпуска доля п/ф-замещения смещается.
- **Порог сверки.** Из-за движения запасов сверка — приблизительная. Нужно
  зафиксировать разумный порог (напр. ≤ несколько %) и не трактовать штатное
  расхождение как баг.
- **Валидация `DEC_SCHEMA`.** Новые позиции (план, время сборки, смены, персонал,
  станки, R&D) выведены по ОДНОМУ файлу — обязательна проверка по всем 4 текущим
  отчётам, как исходная схема (позиции формы стабильны, но проверить).
- **Спот-цена и yield остаются экзогенными.** Заказ сырья (спот/3м/6м) задаёт
  объём закупки, не рыночную цену → цена сырья калибруется по истории, как сейчас.
- **Совместимость чисел.** Числа `cost` изменятся (объём и время сборки) —
  существующие тесты/ожидания на старые значения нужно обновить.
- **Труд-пул не проспективен (v1).** Зарплаты — фикс-пул из отчёта; при крупной
  правке планового объёма (сверх текущих мощностей) труд недо-нагружается, пока
  правка не увеличит число смен. Within-capacity и режим сверки — корректны.
