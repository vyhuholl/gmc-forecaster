## Context

`gmc-forecaster` прогнозирует спрос по 9 ячейкам (3 продукта × 3 канала), но не
считает затраты. Данные затрат уже присутствуют в отчёте на двух именованных
листах, которые парсер игнорирует (читает только плоский лист `W`):

- `Resources and products` — производство: произведено/брак/полуфабрикаты по
  продуктам, часы сборщиков (доступно/отработано/болезни), станки (эффективность,
  доступно/отработано часов), закупки/использование сырья, отгрузки по каналам.
- `Financial statements` — деньги: накладные расходы, P&L (себестоимость
  производства и реализации: зарплаты сборщиков/механиков, эксплуатация станков,
  ОТК, транспорт, амортизация), закупки сырья/полуфабрикатов в единицах и деньгах.

Реверс-инжиниринг (11 отчётов) подтвердил ключевые нормативы и структуру:
`сырьё_использ = Σ_p (произведено_p − полуфабрикаты_p) × {p1:1, p2:2, p3:3}` —
тождество выполняется до единицы; сборка ≈ 100/150/300 мин/ед. yield-факторы
трендовые (брак 3.2→2.2 %, болезни 84→281 ч, загрузка станков 70→96 % за 4 кв.).

Ограничения проекта: Python ≥3.14, `mypy --strict`, ruff line-length 79,
комментарии/вывод на русском, весь код через `uv run`, движок чтения Excel —
`engine="calamine"` (читает и BIFF `.xls`, и strict-OOXML `.xlsx`).

## Goals / Non-Goals

**Goals:**

- Полная себестоимость (absorption) и contribution margin на единицу по 9 ячейкам.
- Калибровка нормативов расхода и yield-факторов на истории (гибрид: табличные
  нормативы как структурная модель + инференс-фоллбэк; yield — EWMA по серии).
- Двухступенчатое разнесение затрат (продукт-пул → канал; канал-пул → продукт;
  overhead → по выручке) с явными, названными в спеке правилами.
- Аддитивная интеграция: новый модуль `cost`, подкоманда `cost`, колонки прибыли
  в `forecast`; `backtest` не трогаем.

**Non-Goals:**

- Оптимизация решения (подбор цен/выпуска под макс. прибыль) — только оценка.
- Планирование производства и мощностей как отдельный решатель (используем
  прогноз спроса + желаемый запас как вход для объёмов, но не оптимизируем).
- Точное воспроизведение бухгалтерской COGS с потоками запасов — считаем
  *экономическую* себестоимость решения снизу вверх, P&L используем для сверки.
- Прогноз самих затрат конкурентов (только своя компания).

## Decisions

**1. Себестоимость строим снизу вверх, а не из строки COGS P&L.**
Строка «Затраты на пр-во и реал. пр.» смешивает движение запасов (остатки
начало/конец), поэтому не отражает маржинальную стоимость решения. Считаем
per-unit из расхода ресурсов × цены ресурсов. P&L — только для валидации
(сверка суммарной сметы с фактом). Альтернатива (разложить строку COGS) отвергнута:
искажается складскими свингами.

**2. Нормативы и ставки затрат — авторитетные константы мануала; инференс —
только валидация.** Все технормативы и фиксированные ставки затрат извлечены из
онлайн-справочника GMC (`dl.globalmanager.ru/sprav/content/tables.htm`, домен с
истёкшим TLS → доступ `curl -k`) и перечислены в Приложении. Ключевое (Табл. 5):
изготовление деталей (станко) 60/75/120 мин, сборка 100/150/300 мин, сырьё 1/2/3
шт на ед. Полуфабрикат замещает станко-этап (не сборку): `машино-часы =
Σ_p (произведено_p − п/ф_p) × {60,75,120} / эффективность` — сверено на 4
кварталах, сходится до 0.06 % (Q4: 4966 vs факт 4963). Наивная регрессия
`материал ~ выпуски` даёт мусор (коэф. p3 < 0) из-за подстановки п/ф — поэтому
инференс МНК демотирован до **sanity-check** нормативов, не источника. Из отчёта
берём только рыночные/решенческие величины: спот-цену сырья (Табл. 4, в отчёте),
зарплаты сборщиков/механиков (ставка — решение), комиссию дистрибьюторов, цены.

**3. yield-факторы — EWMA по серии, а не последний квартал.**
Брак/болезни/эффективность трендовые. EWMA (свежесть важнее) устойчивее точечной
оценки и точнее долгого среднего. Три канала влияния: брак → делить на годные;
болезни → фикс. зарплата на меньший выход (конверсия/ед ↑); эффективность станков
→ больше станко-часов/ед. Один квартал → EWMA вырождается в точку (без ошибки).

**4. Двухступенчатое разнесение — фундаментально, т.к. производство по ПРОДУКТУ,
продажа по КАНАЛУ.** «Себестоимость ячейки» — сконструированное, а не наблюдаемое
число. Этап 1: продукт-затраты (материал+п/ф — переменные; конверсия — фикс-пул по
сборщико-минутам) → в ячейку по единицам `продукт→канал` (из отгрузок).
Этап 2: канал-пул → на продукты внутри канала по единицам. Overhead (маркетинг +
прочие непроизв.) → по выручке ячейки. Драйверы зафиксированы в спеке.

**5. Contribution margin = цена − материал − п/ф − комиссия дистрибьютора.**
Конверсия сборки — полу-фикс (штат нанимается на квартал), в CM не входит. CM —
для решений о ценах/наборе (фиксы невозвратны); полная — для отчётности. Оба
показателя выдаём всегда.

**6. Объёмы производства для forward-сметы = прогноз спроса + желаемый запас.**
Спрос по ячейке (продукт×канал) агрегируется в выпуск по продукту
(`Σ_каналов`), затем себестоимость продукта разносится обратно на каналы по
единицам. Желаемое изменение запаса — параметр (дефолт 0: производим = спрос).

**6a. Производственные фиксы пересобираются по СМЕНАМ, а не берутся прошлым
фактом.** Новое решение (выпуск) меняет нужные машино-часы → минимальное число
смен → штат механиков и техосмотр. Число смен выводится из отчёта:
`смены = ⌈доступные_ст-часы / (станки × 576)⌉` (сверено: Q1=2 при 9×1068, Q2–Q4=1
при 9×576); для forward — минимальные смены под нужные машино-часы. Формулы (m3,
сверены на 4 кв.): техобслуживание = `часы_техосмотра/станок × станки × 85`
(точно); эксплуатация станков = `станки×3500 + смены×12500 + ст-часы×8 + ед×1`
(~99.8 %); зарплаты механиков = `(ст-часы+простой) × ставка × (1+премия_смен) ×
4/станок`, избыток механиков — по мин. ставке (Табл. 17). Ставка зарплат —
решение игрока (лист `Your decisions`); премия смен 0/⅓/⅔ (Табл. 16).

**7. Модуль `cost` и подкоманда `cost` (имя по требованию пользователя, не
`costing`).** Структура зеркалит существующие `forecast`/`backtest`:
`cost(current, train, history, ...)` → df по 9 ячейкам + `df.attrs`. CLI —
третья подкоманда с `--out` → CSV.

## Risks / Trade-offs

- **Инференс расхода нестабилен (коллинеарность выпусков, простой-конфаундер)** →
  Митигация: используем константы мануала (Приложение); инференс — только
  sanity-check (сверка знака/масштаба), не источник нормативов.
- **Цены ресурсов из малой выборки закупок** (например, закуплено 46 ед. сырья) →
  Митигация: сглаживание цены по кварталам серии; при нулевой закупке — перенос
  последней известной цены; юнит-тест на деление на ноль.
- **Позиционные схемы листов могут «съехать» между версиями формы** (как риск для
  `SCHEMA` листа `W`) → Митигация: валидация `RES_SCHEMA`/`FIN_SCHEMA` на всех
  доступных отчётах (RU-2026 + разноязычные старые); сверка суммарной сметы с
  P&L-строкой как контрольная сумма парсинга.
- **Разнесение overhead по выручке зависит от цен решения** (эндогенность) →
  Митигация: это осознанный выбор пользователя; для решений использовать CM, где
  overhead не участвует; полную — как справочную.
- **half-final / history-режим без своих данных** (компания вне 1..8) → Митигация:
  как в `forecast`, деградация: при отсутствии листов затрат — пропуск колонок
  прибыли с предупреждением, `cost` требует валидный отчёт компании 1..8.

## Migration Plan

Аддитивно, без миграции данных. Порядок: (1) расширить `parser.py` схемами и
покрыть тестом-тождеством сырья; (2) `cost.py` — примитивы и разнесение,
самостоятельная подкоманда `cost`; (3) интеграция колонок в `forecast`. Откат —
удаление подкоманды/колонок; `forecast`/`backtest` не затронуты. Каждый шаг
проходит `make validate` (ruff + mypy --strict).

## Приложение: константы мануала GMC
